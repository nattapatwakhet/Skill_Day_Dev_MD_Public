# Controlled Operation Skill

มาจาก [`work_flow_skill.md`](./work_flow_skill.md) ขั้น [3.2] — ใช้เมื่องานต้องดู/คุมหน้าจอหรือคุมเครื่อง user

"การทำงานแบบควบคุม" — กฎว่าก่อนจะลงมือทำอะไรที่แตะเครื่อง user (ดูจอ คุมจอ รันคำสั่ง แก้/ลบไฟล์ ใช้เบราว์เซอร์)
**ต้องเช็คสิทธิ์ก่อนเสมอ** ว่ามีสิทธิ์หรือยัง แล้วทำตามลำดับควบคุม

> ไฟล์รวบรวมสิทธิ์ทั้งหมด = [`permission_skill.md`](./permission_skill.md) — จะทำอะไรเปิดดูไฟล์นั้นว่า "มีสิทธิ์ยัง"
> งาน Docker / docker-compose โดยเฉพาะ = [`docker_skill.md`](./docker_skill.md) (ยังว่าง — ระหว่างนี้ใช้หลัก "รันคำสั่งบนเครื่อง user" ด้านล่าง)

---

## วงจรการทำงานแบบควบคุม (ทำตามทุกครั้ง)

```
1. จะทำอะไร  →  เปิด permission_skill.md เช็คว่างานนี้ต้องใช้สิทธิ์อะไร / มีหรือยัง
2. ยังไม่มีสิทธิ์  →  ขอสิทธิ์ให้ครบในครั้งเดียวตามเครื่องมือที่ runtime รองรับ
3. บอก user ก่อน  →  "จะทำ X ที่ Y" แล้วรอ ok
4. ลงมือทำ        →  ทำในขอบเขตสิทธิ์ที่ได้ ห้ามเกิน
5. เสร็จ          →  คืนจอให้ user + รายงานผลสั้นๆ
```

**กฎเหล็ก 4 ข้อ — ขอ user ก่อนทุกครั้ง:**

1. **ขอก่อนดูหน้าจอ** — ห้ามถ่าย screenshot เองโดยพลการ
2. **ขอก่อนควบคุม** — ห้ามคลิก/พิมพ์/คุมแอปเองโดยพลการ เสร็จแล้ว**คืนจอทันที**
3. **ขอก่อนลบ** — ห้ามลบไฟล์ user เอง (ยกเว้นไฟล์ขยะที่ Claude สร้างเอง)
4. **ขอก่อนงานที่ย้อนไม่ได้** — ส่งข้อความ/ดาวน์โหลด/กดยืนยัน/publish

---

## ลำดับการคุมจอ (operating procedure)

1. บอก user ว่าจะดู/จะคุมอะไร → รอ ok
2. `open_application "<แอป>"` ให้แอปมาอยู่หน้าสุดก่อน (กัน browser/แอปอื่นแย่ง focus จนคลิกพลาด)
3. ทำงาน (คลิก/พิมพ์/ดับเบิลคลิก) — ใช้ `computer_batch` รวบหลายสเต็ปในครั้งเดียวเมื่อคาดผลได้ เร็วกว่า
4. ถ่าย screenshot ปิดท้ายเพื่อตรวจผล
5. **คืนจอให้ user**

> ระวัง: ระหว่างเรียกแต่ละครั้ง แอปอื่นอาจเด้งมาแย่ง frontmost ทำให้คลิกโดน error "granted at tier read/click" → แก้โดย `open_application` แอปเป้าหมายใหม่แล้วลองอีกครั้ง

---

> ถ้างานคือ **รันคำสั่ง/สคริปต์บนเครื่อง user** (เช่น Terminal พิมพ์ไม่ได้ ต้องใช้สคริปต์ `.command`) →
> นั่นเป็นอีกขั้นแยกใน flow ดู [`work_flow_skill.md`](./work_flow_skill.md) ขั้น [5] → [`terminal_skill.md`](./terminal_skill.md)

## คุมเบราว์เซอร์

- ถ้า runtime มี browser automation tool ให้ใช้เครื่องมือนั้นก่อนการคุมจอแบบคลิกพิกัด
- นำทาง/คลิก/กรอกฟอร์มด้วย DOM automation เมื่อทำได้ เพราะแม่นกว่าคลิกตำแหน่ง
- ถ้าต้องใช้ browser session ของ user ให้เช็คก่อนว่ามี browser/extension/connector ที่พร้อมใช้งาน
- ความปลอดภัยลิงก์: อย่าคลิกลิงก์จากอีเมล/แชตที่ไม่รู้จัก ตรวจ URL จริงก่อนเสมอ

---

## ทำไมต้องอ้อม (sandbox แยกจากเครื่อง user)

- `sandbox bash` เป็นเครื่อง Linux แยกต่างหาก — ไม่มี Docker, เข้า `localhost` ของ user ไม่ได้
- `localhost` ใน sandbox = ตัว sandbox เอง ไม่ใช่ Mac ของ user
- จะแตะ docker/localhost จริง ต้องให้คำสั่งไปรัน "บนเครื่อง user" ผ่านทางที่ runtime รองรับ เช่น สคริปต์ที่ user ตรวจได้, user รันเอง, หรือ browser/terminal automation ที่ได้รับอนุญาต
