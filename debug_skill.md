# Debug Skill — วินัยการแก้บั๊ก + บันทึกหลังแก้

มาจาก [`code_skill.md`](./code_skill.md) — ใช้เมื่องาน code เป็นการ **debug / หาบั๊ก / แก้ของที่พัง**
(ปรับหลักการมาจากแนวคิด debug-mantra + post-mortem)

> เป้าหมาย: อย่าเดาแล้วแก้มั่ว ทำตาม 4 ขั้นตามลำดับ เสร็จแล้วบันทึก root cause ไว้กันลืม

---

## 4 ขั้นของการ debug (ทำตามลำดับ ห้ามข้าม)

```
1. reproduce ให้ได้ก่อน   → ทำซ้ำบั๊กได้แน่นอนไหม
2. หา fail path           → บั๊กพังตรงไหน อะไรกันไม่ให้พัง
3. ลองหักล้าง hypothesis  → อะไรจะพิสูจน์ว่าสมมติฐานนี้ผิด
4. ทุก run คือ breadcrumb → จดทุกการทดลอง เทียบกันทั้งหมด
```

### 1. Reproduce ให้ได้ก่อน

- **ทำซ้ำได้แน่นอน** → จับ step/input/env เป็น artifact ที่รันซ้ำได้ (failing test, curl, CLI, replay)
- **flaky (เป็นบ้างไม่เป็นบ้าง)** → ยังแก้ไม่ได้ ต้องเพิ่มอัตราการเกิดก่อน (วน loop, ยิงพร้อมกัน, ใส่ stress, แคบ timing) — 50% แก้ได้ 1% แก้ไม่ได้
- **ทำซ้ำไม่ได้เลย** → **หยุด** บอก user ตรงๆ ขอ env/log/artifact หรือขออนุญาต instrument ก่อน **ห้ามเดา hypothesis**

> เป้า: สัญญาณ pass/fail เร็ว (1–5 วิ) และ deterministic — pin เวลา, seed RNG, freeze network

### 2. หา fail path

หาว่าโค้ดพังตรงไหน และอะไรกันไม่ให้พัง ทำไล่จากเบาไปหนัก:

1. **Debugger ก่อน** — ถ้า env รองรับ attach แล้ว step ไปจุดพัง (1 breakpoint ดีกว่า 10 log)
2. **Source trace + ไล่ knob** — ถ้าไม่มี debugger: ไล่ code path ทั้งเส้น แล้วลิสต์ทุกตัวแปรที่มีผล (config, env, flag, branch, input shape, timing, concurrency) แล้วลอง**พลิกทีละตัว**
3. **ใส่ log ในโค้ด** — ถ้า knob ข้างนอกขยับบั๊กไม่ได้ ใส่ log/dump state ตรงจุดต้องสงสัย ติด prefix เฉพาะ (เช่น `[DBG-a4f2]`) ให้ลบทีหลังด้วย grep ครั้งเดียว

> สำหรับ Flutter ใช้ `PrintUtil().printDebug(...)` ไม่ใช่ `print(...)` (ดู `flutter_skill.md`)
> ถ้าต้องรันคำสั่ง/ดู log บนเครื่อง user → ดู [`terminal_skill.md`](./terminal_skill.md)

### 3. ลองหักล้าง hypothesis

เมื่อเจอผู้ต้องสงสัย → **ตรวจก่อนเทส**:

- มันอธิบายอาการได้ครบทั้งเส้นไหม เดินดูจริง
- อะไรคือ**ข้อพิสูจน์**ที่ง่ายที่สุด อะไรคือ**ข้อหักล้าง**ที่สะอาดที่สุด
- **รันข้อหักล้างก่อน** — ถ้า hypothesis รอด แปลว่าจริง ถ้าตาย ก็รอดจากการไล่ผี
- ตั้ง hypothesis 3–5 อันจัดอันดับ อย่ายึดอันแรกอันเดียว

### 4. ทุก run คือ breadcrumb

จด **ledger** ทุกการทดลองในรอบนี้ แต่ละบรรทัด: เปลี่ยนอะไร / เกิดอะไร / ตัดอะไรออก-เข้าได้

- hypothesis ใหม่ต้องอธิบาย**ทุก**สิ่งที่เคยเห็น ไม่ใช่แค่ล่าสุด
- ถ้ามี run เก่าขัดแย้ง → hypothesis ผิดหรือไม่ครบ
- ถ้าลังเล ออกแบบ **1 การทดลอง** ที่ผลออกมาแล้วชี้ขาด แล้วรันอันนั้น

## กฎการทำงาน

- ทำ 4 ขั้น**ตามลำดับ**: ห้ามเสนอ fix ก่อนมี repro / ห้ามเทส hypothesis ก่อนแคบ fail path / ห้ามฟันธงก่อนลองหักล้าง / ห้ามสรุปถูกก่อนเทียบ breadcrumb ครบ
- จับได้ว่ากำลังจะ fix โดยไม่มี repro → กลับไปขั้น 1
- แก้ในขอบเขตที่ user ขอ (ดู scope ใน [`code_skill.md`](./code_skill.md))

## บันทึกหลังแก้บั๊ก (post-mortem แบบสั้น)

เมื่อแก้เสร็จ + verify แล้ว ให้บันทึกไว้ (กันลืม + ให้คนอื่นตามได้) — **ไม่แก้เสร็จจริงอย่าเขียน**

ต้องมีครบ 4 อย่างก่อนเขียน: repro ที่เชื่อถือได้ / รู้ root cause จริง (ไม่ใช่เดา) / มี fix (PR/commit) / verify แล้ว (repro เดิมผ่าน)

หัวข้อที่ควรมี:

1. **สรุป** — พังอะไร (ในมุมผู้ใช้/งาน) แก้ด้วยอะไร 1 ประโยค + ลิงก์ PR/ticket
2. **อาการ** — ที่เห็นจริง (error/log/พฤติกรรม) อย่าพาราเฟรส
3. **Root cause** — กลไกบั๊กจริง ใส่ชื่อ function/ไฟล์/บรรทัดได้เต็มที่ เดินเหตุทั้งเส้น
4. **ทำไมอาการออกมาแบบนั้น** — โยง root cause → อาการ
5. **Fix** — แก้อะไร ทำไมแก้ที่ต้นเหตุ ไม่ใช่กลบอาการ
6. **ทำไมหลุดมาได้** — CI ไม่มีเทสเส้นนี้ / โค้ด latent / workload ไม่เคยแตะ / fix เก่ากลบไว้ (blameless — เล่าช่องโหว่ ไม่โทษคน)
7. **Validation** — รู้ได้ยังไงว่าแก้จริง (เทสไหนผ่าน, workload ไหนรันได้) บอกขอบเขตที่เทสตามจริง อย่าโม้ว่าครอบคลุมกว่าที่ทำ

> ถ้าต้องสรุปเรื่องนี้ส่งให้คนไม่ใช่ dev (หัวหน้า/PM) → ส่งต่อ [`report_skill.md`](./report_skill.md)
> ถ้าเจอแพตเทิร์น/กฎ debug ใหม่ → บันทึกกลับไฟล์นี้ (ดูท้าย [`main_skill.md`](./main_skill.md))

## บทเรียน: อาการข้ามชั้น → ขอ artifact ชี้ขาดก่อนไล่แก้ปลายเหตุ

อาการที่ผิวเป็น frontend แต่ข้อมูลมาจาก API/DB อย่าด่วนแก้โค้ด frontend

กฎเพิ่ม:
1. **ขอ "response จริงของ endpoint" เป็น artifact แรก** (Postman/Network) ก่อนแตะโค้ด frontend
   — มันชี้ชั้นที่พังทันที
2. ดูตัวเลขใน response ให้ครบ ไม่ดูแค่ `data` — `total`/count ต่างๆ เป็นเบาะแส
   (เช่น `data:[]` แต่ count = จำนวนแถวตาราง **อื่น** → ชี้ว่ายิงผิด table/route)
3. ถ้า "โค้ด source ถูกแต่ runtime พัง" → สงสัย layer ระหว่างทางก่อน (routing, filter,
   container รันโค้ดเก่า, ต่อ DB ผิดตัว) อย่าด่วนแก้ model

วินัยบันทึก: **อัปเดต skill เองทันทีหลังจบงาน ไม่รอให้ user สั่ง**
