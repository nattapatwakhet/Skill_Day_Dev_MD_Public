# Report Skill — สรุปงาน eng ให้คนไม่ใช่ dev อ่าน

มาจาก [`work_flow_skill.md`](./work_flow_skill.md) ขั้น [8] — ใช้เมื่อต้อง **รายงาน/สรุปงาน** ที่เป็นเรื่องเทคนิค
ให้คนที่ไม่ได้เขียนโค้ด (หัวหน้า, PM, ทีมอื่น) และ**ปรับรูปตามช่องทาง**ที่จะส่ง
(ปรับหลักการมาจากแนวคิด management-talk)

> หลัก: คนอ่านเข้าใจชื่อ product/feature แต่ไม่อ่านโค้ด — บอกว่า *สถานะอะไร แปลว่าอะไรกับผู้ใช้ ใครดูแล ต่อไปทำอะไร*

---

## เก็บ / ตัด / แปล

- **เก็บ** — ชื่อ product/feature/โมดูล, เลข ticket/PR, ชื่อ workload/ลูกค้า (เป็นสะพานให้คนติดตามงานได้)
- **ตัด** — ชื่อ function/ไฟล์/บรรทัด/commit SHA/ชื่อ env var/ศัพท์ data-structure ภายใน (คนอ่านเอาไปทำอะไรไม่ได้)
- **แปล** — กลไกเป็นเหตุ-ผลภาษาคน 1–2 ประโยค โดยไม่โกหก (race ก็ยังเป็น race, regression ก็ยังเป็น regression)
- **อย่าตัดเกิน** — คนระดับหัวหน้าสาย eng อ่านศัพท์ concept ออก (race condition, sync, regression, workaround) เส้นแบ่งคือ *concept มีจริงและเกี่ยว* (เก็บ) vs *นี่คือชื่อ function/ไฟล์* (ตัด)
- **เน้น** active voice ประโยคสั้น: "เจอบั๊กแล้ว Alex แก้อยู่ PR รออยู่" ดีกว่าประโยคยาวอ้อม

**เลี่ยง:** hedging ที่ไม่ใช่ hedge ("น่าจะ/ดูเหมือน"), พูดของชัดๆ ซ้ำ, สั่งหัวหน้าว่าต้องทำอะไร, รายละเอียด process debug (bisect/gdb)

---

## รูปตามช่องทาง (เนื้อเดียว เปลือกต่าง)

### รายงานเต็ม / ticket
บล็อกมีหัวข้อ bold สแกนง่าย — ใช้เท่าที่เข้า: **สถานะ/TL;DR** (1 บรรทัด หยุดตรงนี้ก็ได้คำตอบ), **ผลกระทบ** (ใครเจอ เห็นอะไร), **พังอะไร** (กลไกภาษาคน), **ทำไมตอนนี้/หลุดมาได้** (ถ้าจะโดนถาม), **คนดูแล** (คน+ทีม+ลิงก์เดียว), **ขั้นต่อไป**, **workaround** (ถ้าลูกค้าโดนอยู่)

### Slack
ข้อความเดียว ไม่ผนังตัวอักษร: **TL;DR bold** 1 บรรทัด + 2–4 bullet สั้น (ผลกระทบ/คนดูแล+ลิงก์/ขั้นต่อไป) ลิงก์ฝังในบรรทัด ไม่มีทักทาย/ลงท้าย (~ไม่เกิน 80 คำ)

### Standup note
1–3 บรรทัด front-load กริยา: "*\<สถานะ\> \<อะไร\>. \<คนดูแล\>. \<ต่อไป\>.*" ไม่ bullet ไม่หัวข้อ

### Email
**Subject** = TL;DR เป็นวลีนาม / ทักทายตาม register / body เป็นย่อหน้าไหลๆ (ไม่ต้องหัวข้อ bold) 2–3 ย่อหน้าพอ / ลงท้ายด้วยจุดที่ต้องการการตัดสินใจจากผู้รับ (ถ้ามี)

### Talking-points (พูดในที่ประชุม)
bullet สั้น 1 clause เรียงตามลำดับที่จะพูด ใส่ตัวเลข/key ที่จะอ้างลงใน bullet เลย ไม่ต้องร้อยประโยค

## flow การทำงาน

1. **เช็กช่องทางก่อน** ถ้ายังไม่บอก ถาม 1 คำถามสั้นแล้วหยุด ("JIRA, Slack, standup หรือ email?")
2. **ร่างเป็นบล็อกเดียว** จัดรูปตามที่ช่องทางนั้นจะ render
3. **default = print ให้ user ก๊อปเอง** — **ห้ามส่ง/โพสต์แทน** ทุกช่องทาง (ถ้าจะโพสต์จริงต้องขอ user ก่อน ดู [`permission_skill.md`](./permission_skill.md))

## สรุปงานรายวัน/รายเดือนสำหรับสไลด์

ถ้า user ถามว่า "วันนี้ทำอะไรบ้าง", "เดือนนี้ทำอะไรไปบ้าง", หรือบอกว่าจะเอาไปใส่สไลด์:

- ใช้ [`work_summary_skill.md`](./work_summary_skill.md) เพื่อดึงข้อมูลจาก `skill/day_dev_public/memory/WORK_LOG.md`
  และ `skill/day_dev_public/memory/PROGRESS.md`
- เรียบเรียงเป็นผลลัพธ์/ผลกระทบ ไม่ใช่รายชื่อไฟล์ที่แก้
- รวมงานย่อยที่เกี่ยวกันเป็นหัวข้อใหญ่ เช่น "ยกระดับคุณภาพโค้ด React/TypeScript"
- ใส่คำสั่งตรวจผ่านเฉพาะที่มีผลกับความน่าเชื่อถือ เช่น `npm run lint`, `npm run typecheck`, `npm run build`
- แยก "ประเด็นต่อเนื่อง" ออกจาก "งานเสร็จแล้ว"
- หัวข้อ "ประเด็นต่อเนื่อง" ต้องรวม 3 อย่างให้ครบถ้ามี:
  งานที่ยังเหลือ/Next, blocker/คำถามค้าง, และคำแนะนำที่ยังไม่ได้ทำตาม

## กฎ

- **ห้ามแต่งข้อมูล**ให้รายงานสวย ถ้าต้นทางบอก "ยังไม่รู้ root cause" รายงานก็ต้องบอกแบบนั้น
- **ห้ามตัด** เลข ticket/PR/ชื่อลูกค้า ตอน de-jargon (เป็นสะพานติดตามงาน)
- **ห้ามเดาคนดูแล** ถ้าต้นทางไม่ระบุ ถาม user
- **ไม่ advocacy** — รายงานคือสถานะ ไม่ใช่ข้อเสนอ ถ้า user อยากได้ memo เสนอแนะ ค่อยเปลี่ยนโหมด
- ถ้าต้นทางเป็น post-mortem วิศวกรรม ดึงมาจาก [`debug_skill.md`](./debug_skill.md) แล้ว reframe — engineering truth อยู่ที่นั่น report แค่แปลขึ้นบน
