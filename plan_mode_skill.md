# Plan Mode — วางแผนก่อนลงมือ (อ่านก่อนทำงานใหญ่)

มาจาก [`work_flow_skill.md`](./work_flow_skill.md) ขั้น **[7] ก่อนลงมือ** — ใช้เมื่องาน **ใหญ่/ซับซ้อน/แตะหลายไฟล์/ระบบสำคัญ**
(แก้ระบบ login, refactor, แตะหลาย layer, งานที่พังแล้วเจ็บ)

แก่น: **AI วางแผนก่อน ยังไม่แตะไฟล์** → user อ่านแผน → อนุมัติ → ค่อยลงมือทีละไฟล์
ต่างจาก [`review_skill.md`](./review_skill.md): review = ตรวจของที่ทำแล้ว · plan mode = ออกแบบลำดับก่อนทำ

> งานเล็ก/ชัด/แตะไฟล์เดียว → ข้ามได้ ลงมือเลย (อย่าวางแผนเกินจำเป็น — ดู [`ponytail_skill.md`](./ponytail_skill.md))

---

## เมื่อไหร่เข้า Plan Mode

เข้าเมื่อมี **อย่างน้อย 1 ข้อ**:
- แตะ **หลายไฟล์/หลายฟังก์ชัน** หรือข้าม layer (frontend+backend, model+controller)
- ระบบ **สำคัญ/เสี่ยง** (auth, payment, migration, ลบข้อมูล)
- **ยังไม่รู้สาเหตุชัด** — ต้องสำรวจโค้ดก่อนถึงจะรู้ว่าต้องแก้ตรงไหน
- มี **หลายทางเลือก** ในการแก้ ต้องชั่งน้ำหนัก
- user สั่งตรงๆ ว่า "วางแผนก่อน / อย่าเพิ่งแก้ / plan mode"

ไม่เข้าเมื่อ: แก้ typo, เปลี่ยนค่า config, เพิ่ม log, งานไฟล์เดียวที่เห็นชัดว่าต้องทำอะไร

---

## ลำดับ Plan Mode (4 ขั้น)

### 1. รับโจทย์ + เป้าหมาย
- สรุปปัญหา + ผลลัพธ์ที่อยากได้เป็น 1–2 ประโยคด้วยคำตัวเอง
- ถ้าโจทย์ยังคลุม → ถามให้ชัดก่อน **ยังไม่สำรวจโค้ด**

### 2. สำรวจโค้ด (อ่านอย่างเดียว — ห้ามแก้)
- เปิด [`project_structure_skill.md`](./project_structure_skill.md) รู้ว่าโปรเจคไหน ไฟล์อยู่ layer ไหน
- ไล่ไฟล์/ฟังก์ชันที่เกี่ยวกับปัญหา trace flow จริง (ยืมท่า Trace จาก [`review_skill.md`](./review_skill.md))
- สรุปว่า "ของจริงตอนนี้ทำงานยังไง" + "จุดที่ต้องแตะมีตรงไหนบ้าง"

### 3. เขียนแผนเป็น checklist ให้ user ตรวจ
รูปแบบ — ต่อหนึ่งไฟล์/หนึ่งจุดที่จะแก้:

```
[ ] path/to/file.ext → ฟังก์ชัน foo()
    แก้อะไร: ...
    ทำไม: ... (โยงกลับเป้าหมาย)
[ ] path/to/other.ext → ...
```

ปิดท้ายแผนด้วย:
- **ลำดับการแก้** + เหตุผล (อะไรต้องก่อน/หลัง, dependency)
- **ทางเลือก** ถ้ามี (ทางA/ทางB ข้อดีข้อเสีย + แนะนำอันไหน เพราะอะไร)
- **ความเสี่ยง / สิ่งที่จะกระทบ** (caller อื่น, data, migration)
- **จะตรวจยังไง** ตอนเสร็จ (ตาม [`work_flow_skill.md`](./work_flow_skill.md) ขั้น [8])

> ก่อนเสนอแผน รัน Intent pass จาก [`review_skill.md`](./review_skill.md): "มีวิธีง่ายกว่า/ไม่ต้องทำไหม"
> ถ้าเจอทางที่ง่ายกว่า → เสนอในแผน นี่คือ output ที่มีค่าสุด

### 4. รออนุมัติ → ลงมือทีละไฟล์
- **หยุดรอ user ตอบ ok** ก่อนแตะไฟล์จริง
- อนุมัติแล้ว → แก้ **ทีละไฟล์ตามแผน** จบไฟล์ไหนสรุปสั้นๆ ว่าแก้อะไร
- แผนเปลี่ยนระหว่างทาง (เจอของไม่คาด) → บอก user + อัปเดต checklist ก่อนไปต่อ

## เลือกโหมดคิด vs โหมดทำ (Opus / Sonnet)

ใช้คนละแรง คนละจังหวะ:

| จังหวะ | ใช้ | ทำอะไร |
|---|---|---|
| **คิด/วางแผน** (ขั้น 1–3) | Opus | วิเคราะห์ระบบใหญ่ ชั่งทางเลือก ออกแบบลำดับ |
| **ลงมือ** (ขั้น 4) | Sonnet | เขียน/แก้ไฟล์ตามแผนที่อนุมัติแล้ว เร็ว/ประหยัดกว่า |

> หลักง่ายๆ: **งานยาก/ระบบใหญ่ → วางแผนก่อน (Opus) พอแผน ok → แก้ทีละส่วน (Sonnet)**
> งานเล็กชัด ข้ามไป Sonnet ลงมือเลย ไม่ต้องเข้า plan mode

## prompt ที่ใช้สั่งได้เลย

เริ่ม:
```
เข้า Plan Mode ก่อน อย่าเพิ่งแก้ไฟล์
อ่านโครงสร้างโปรเจคนี้ แล้ววิเคราะห์ว่าเพื่อแก้ [ใส่ปัญหา]
ต้องแก้ไฟล์ไหนบ้าง ทำไมต้องแก้ ลำดับการแก้ควรเป็นยังไง
สรุปเป็น checklist ให้ตรวจก่อน
```

อนุมัติ:
```
โอเค ทำตามแผนได้ แก้ทีละไฟล์ และอธิบายสั้นๆ ว่าแก้อะไร
```

---

## กฎ

- **ห้ามแก้ไฟล์ก่อนอนุมัติ** — plan mode = อ่าน+เขียนแผน เท่านั้น
- **แผนต้องลงมือได้** — ทุกข้อชี้ไฟล์/ฟังก์ชันจริง ไม่ใช่ "ปรับ logic ให้ดีขึ้น" ลอยๆ
- **เสนอวิธีง่ายกว่าเสมออย่างน้อย 1 รอบ** ก่อนเสนอแผนยาว
- **ทีละไฟล์ตอนทำ** — ไม่ถล่มแก้ทุกไฟล์รวด user ตามไม่ทัน
