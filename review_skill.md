# Review Skill — รีวิวงานแบบคนนอก (ทุกโดเมน)

มาจาก [`work_flow_skill.md`](./work_flow_skill.md) — ใช้เมื่อต้อง **รีวิว/ตรวจ/ขอความเห็นที่สอง** ต่อ "อะไรก็ได้":
plan, PR, โค้ด, เอกสาร, ดีไซน์, งานวิจัย, ข้อเสนอ — **ใช้ได้ทุกโดเมน ไม่ใช่แค่ code**
(ปรับหลักการมาจากแนวคิด scrutinize)

> แก่นใช้ได้กับทุกงาน: มองแบบคนนอก → หาวิธีง่ายกว่า → ไล่ดูของจริงทั้งเส้น → verify ว่าทำได้ตามที่อ้าง
> ตัวอย่าง/คำศัพท์ด้านล่าง (diff, call path, file:line) เป็นเคส **code** — งานโดเมนอื่นเทียบเป็น:
> เอกสาร→ไล่ทีละหัวข้อ/ข้ออ้าง, ดีไซน์→ไล่ flow ผู้ใช้, วิจัย→ไล่ข้อมูล→ข้อสรุป

ใช้ได้ 2 จังหวะ:
- **ก่อนลงมือ** (ขั้น [7] ใน work_flow) — รีวิว plan ว่าควรทำแบบนี้ไหม มีวิธีง่ายกว่าไหม
- **ตอนตรวจงาน** (ขั้น [8]) — รีวิวผลงานที่ทำแล้วว่าทำได้จริงตามที่อ้างไหม (code = diff/path, งานอื่น = เนื้อหา/ข้ออ้าง)

---

## ท่ายืน

- **คนนอก** — ลืมว่าใครเขียน/ทำไมเขาคิดว่าถูก อ่าน artifact แบบเย็นๆ
- **ดูทั้งเส้น ไม่ใช่แค่ diff** — diff เป็นจุดเริ่ม ไม่ใช่ขอบเขต ตาม call path จริง
- **กระชับ ลงมือได้ มีเหตุผล** — ทุกข้อบอก *แก้อะไร / ทำไม / หลักฐานอะไร* ไม่เล่า diff ซ้ำ

---

## ขั้นตอน (ตามลำดับ)

### 1. Intent — จริงๆ มันจะทำอะไร

- บอกเป้าหมายเป็น 1 ประโยคด้วยคำตัวเอง ถ้าบอกไม่ได้ = artifact ยังไม่ชัด → บอกแล้วหยุด
- ถาม: **มีวิธีง่ายกว่า/เล็กกว่า/สวยกว่าที่ได้ผลเดียวกันไหม?**
  - ไม่ทำเลยได้ไหม (ปัญหาจริงหรือเปล่า)
  - ใช้ของที่มีอยู่แล้วในโปรเจคแทนการเพิ่มของใหม่
  - แก้เล็กที่ได้ 90% ของเป้า ด้วยความเสี่ยง 10%
  - แก้คนละ layer (config แทน code, build แทน runtime)
- ถ้ามีทางที่ดีกว่า → บอกชัดพร้อมเหตุผล **นี่คือ output ที่มีค่าที่สุด** พูดก่อน review ทีละบรรทัด

### 2. Trace — เดิน code path จริง

- ทุกพฤติกรรมที่อ้าง → เดินทั้งเส้น: entry → call sites → branch → state ที่เปลี่ยน → ผลลัพธ์/side effect
- รวมโค้ดที่ไม่ได้แก้สองข้าง diff ด้วย — บั๊กชอบซ่อนตรงรอยต่อ
- plan/design → เดิน flow ที่เสนอเทียบระบบจริง มันไปแตะอะไร สมมติอะไรที่ไม่จริง
- จุดไหน trace แล้วงง (branch ไม่คาด, state ไม่รู้ว่ามี) = สัญญาณ จดไว้

### 3. Verify — ทำได้จริงตามที่อ้างไหม

- **code path ที่เพิ่ง trace ให้ผลตามที่อ้างจริงไหม** เดินให้เห็น "อ้าง X → path A→B→C → ที่ C เห็น [...] → ดังนั้นจริง/ไม่จริง"
- **input/state อะไรทำให้พัง** — edge case, concurrent, error path, null/empty/ใหญ่, ลำดับ
- **มันแอบเปลี่ยนอะไร** — perf, error semantics, contract ของ caller อื่น, format บน disk/wire
- **เทสยังไง** — เทสแตะ path ที่ trace จริงไหม หรือผ่านเพราะ mock บัง / เทสแค่ happy path

### 4. รายงาน

หนึ่ง finding หนึ่งบล็อก เรียงตามความรุนแรง (blocker → major → nit):

- **Finding** — 1 ประโยค ชี้เฉพาะ อ้าง `file:line` ถ้าได้
- **ทำไมสำคัญ** — ผลกระทบ ไม่ใช่หลักการ
- **หลักฐาน** — step ที่ trace / input ที่ทำให้เห็น
- **แก้ยังไง** — รูปธรรม เล็กที่สุด

ปิดท้าย 1 บรรทัด: ship / แก้แล้วค่อย ship / รื้อใหม่ / reject + เหตุผลใหญ่สุด

## Over-engineering pass (รีวิวหาของที่ "ตัดได้")

โหมดเสริม: รีวิวเฉพาะเรื่อง **over-engineering** — หาของที่ลบ/ย่อ/แทนด้วย stdlib ได้
(คู่กับหลักใน [`coding_principles_skill.md`](./coding_principles_skill.md))

ทำได้ 2 ขอบเขต:
- **รีวิว diff/PR** — มองเฉพาะส่วนที่แก้ ว่ามีของเกินไหม
- **audit ทั้ง repo** — สแกนทั้งโปรเจค ออกเป็นลิสต์จัดอันดับว่าควรลบ/ย่ออะไรก่อน (รายงานเฉยๆ ไม่แก้ให้)

มองหา:
- โค้ดที่ reinvent stdlib / ฟีเจอร์ native (เขียนเองทั้งที่ภาษา/แพลตฟอร์มมีให้)
- dependency ที่ไม่จำเป็น (ไม่กี่บรรทัดก็ได้)
- abstraction เก็งกำไร (interface impl เดียว, factory ของชิ้นเดียว, config ค่าที่ไม่เปลี่ยน)
- dead flexibility / boilerplate "เผื่ออนาคต"

หนึ่ง finding หนึ่งบรรทัด: **ตำแหน่ง — ตัดอะไร — อะไรแทน**

> เก็บ shortcut ที่จงใจ (คอมเมนต์ `ponytail:`) เป็น debt ledger — grep หา `ponytail:` ทั้ง repo

## กฎ

- **ห้าม rubber-stamp** — "LGTM" ไม่ใช่ output ถ้าไม่เจออะไรจริงๆ บอกว่า trace/เช็คอะไรไปบ้าง
- **อ้างได้ค่อยพูด** — ทุกข้อเรื่องโค้ดต้องอ้าง path/file/line ห้าม "อาจพังตอน load"
- **แยก "อ้าง" กับ "ยืนยันแล้ว"** — "PR บอกว่า X" กับ "trace แล้วยืนยัน X" คนละอย่าง
- **ต้องถามวิธีที่ง่ายกว่าอย่างน้อย 1 รอบ** เสมอ เว้นแต่ user บอกว่าอย่าแตะ scope
- **อย่ายัด nit เรื่อง style ตอนมีปัญหาเชิงโครงสร้าง** — ขึ้นด้วยเรื่องใหญ่ก่อน
- **ไม่ประจบ ไม่อ้อม** — บอก finding ตรงๆ
