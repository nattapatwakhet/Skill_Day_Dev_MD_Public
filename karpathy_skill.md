# Karpathy Skill — วินัยลดความผิดพลาดเวลา LLM เขียนโค้ด

มาจาก [`coding_principles_skill.md`](./coding_principles_skill.md) — หลักคิด "คิดให้รอบคอบก่อนเขียน"
(ปรับจาก karpathy-guidelines)

> Tradeoff: หลักพวกนี้เอนไปทางรอบคอบมากกว่าเร็ว — งาน trivial ใช้วิจารณญาณได้

---

## 1. คิดก่อนเขียน

**อย่าเดา อย่าซ่อนความสับสน เปิดเผย tradeoff**

ก่อนลงมือ:
- บอก **สมมติฐาน** ออกมาชัดๆ ถ้าไม่แน่ใจ → ถาม
- มีหลายการตีความ → เสนอให้เห็น อย่าเลือกเองเงียบๆ
- มีวิธีง่ายกว่า → บอก กล้าค้านเมื่อมีเหตุผล
- งงตรงไหน → หยุด บอกว่างงอะไร แล้วถาม

## 2. เรียบง่ายก่อน

**โค้ดน้อยที่สุดที่แก้ปัญหาได้ ไม่มีอะไรเก็งกำไร**

- ไม่มีฟีเจอร์เกินจากที่ขอ
- ไม่มี abstraction สำหรับโค้ดใช้ครั้งเดียว
- ไม่มี "ความยืดหยุ่น/configurable" ที่ไม่ได้ขอ
- ไม่มี error handling สำหรับเคสที่เป็นไปไม่ได้
- เขียน 200 บรรทัดแล้วมันควรเป็น 50 → เขียนใหม่

ถาม: "senior จะบอกว่าซับซ้อนเกินไปไหม?" ถ้าใช่ → ทำให้ง่ายลง

## 3. แก้แบบผ่าตัด

**แตะเฉพาะที่ต้องแตะ เก็บกวาดแค่ของที่ตัวเองทำเลอะ**

ตอนแก้โค้ดเดิม:
- ห้าม "ปรับปรุง" โค้ด/คอมเมนต์/ฟอร์แมตข้างเคียง
- ห้าม refactor ของที่ไม่ได้พัง
- ทำตาม style เดิม แม้ตัวเองจะทำอีกแบบ
- เจอ dead code ที่ไม่เกี่ยว → บอก อย่าลบเอง

ตอนการแก้ของเราทำให้เกิด orphan:
- ลบ import/ตัวแปร/ฟังก์ชันที่ **การแก้ของเรา** ทำให้ไม่ถูกใช้
- ห้ามลบ dead code เดิมที่มีอยู่ก่อน เว้นแต่ถูกขอ

เกณฑ์: ทุกบรรทัดที่เปลี่ยน ต้องโยงตรงไป request ของ user

## 4. ทำงานแบบมีเป้าวัดผล

**กำหนดเกณฑ์สำเร็จ แล้ววนจนกว่าจะ verify ได้**

แปลงงานเป็นเป้าที่ตรวจได้:
- "เพิ่ม validation" → "เขียนเทส input ที่ผิด แล้วทำให้ผ่าน"
- "แก้บั๊ก" → "เขียนเทสที่ reproduce บั๊ก แล้วทำให้ผ่าน" (ดู [`debug_skill.md`](./debug_skill.md))
- "refactor X" → "เทสผ่านทั้งก่อนและหลัง"

งานหลายขั้น → บอก plan สั้นๆ พร้อมจุดตรวจ:
```
1. [ขั้น] → ตรวจ: [เช็คอะไร]
2. [ขั้น] → ตรวจ: [เช็คอะไร]
```

เกณฑ์สำเร็จที่แข็งแรง = วนเองได้ / เกณฑ์อ่อน ("ทำให้มันใช้ได้") = ต้องถามตลอด
