# Work Summary Skill — สรุปงานรายวัน/รายเดือนจากบันทึกงาน

มาจาก [`work_flow_skill.md`](./work_flow_skill.md) ขั้น [8] และเชื่อมกับ [`progress_skill.md`](./progress_skill.md)
ใช้เมื่อต้องตอบคำถามแบบ **"วันนี้ทำอะไรไปบ้าง"**, **"เดือนนี้ทำอะไรไปบ้าง"**,
**"เหลืออะไร"**, **"ยังไม่ได้ทำอะไร"**, **"ยังไม่ได้ทำอะไรตามคำแนะนำ"**,
หรือ **"สรุปงานไปใส่สไลด์"** จากหลักฐานที่จดไว้

> `PROGRESS.md` = สถานะงานล่าสุด/งานค้างเพื่อ resume  
> `WORK_LOG.md` = ledger งานที่ทำเสร็จหรือมีความคืบหน้า ใช้สรุปย้อนหลังรายวัน/รายเดือน

---

## ไฟล์บันทึก

เก็บที่:

```text
skill/day_dev_public/memory/WORK_LOG.md
```

ไฟล์นี้เป็น local work memory สำหรับ workspace ที่ติดตั้งชุดนี้

---

## ต้องจดเมื่อไหร่

ทุกครั้งที่จบงานจริงหรือจบก้อนงานที่มีผลลัพธ์ชัดเจน ให้ append entry เข้า `WORK_LOG.md`
หลังอัปเดต `PROGRESS.md` แล้ว

จดให้พอเอาไปทำสไลด์ได้:

- วันที่/เวลา
- โปรเจกต์/ระบบ
- หมวดงาน เช่น `code`, `skill`, `config`, `debug`, `document`, `research`
- ทำอะไร
- ผลลัพธ์/ผลกระทบ
- ตรวจผ่านอะไร
- เหลืออะไร/ข้อเสนอแนะ โดยแยกให้รู้ว่าเป็นงานค้าง, blocker, หรือคำแนะนำที่ยังไม่ได้ทำตาม

---

## รูปแบบ `WORK_LOG.md`

```markdown
# WORK LOG — day_dev_public

## 2026-07

### 2026-07-09

#### <เวลา> — <หัวข้องานสั้น>
- ระบบ/โปรเจกต์: <ชื่อโปรเจกต์หรือ skill>
- หมวด: <code/skill/config/debug/document/research/...>
- ทำไปแล้ว:
  - <bullet สั้น>
- ผลลัพธ์:
  - <user/business impact ไม่ใช่รายละเอียด code ล้วน>
- ตรวจแล้ว:
  - <คำสั่ง/หลักฐานที่ผ่าน หรือ "ไม่ได้รัน">
- เหลือ/แนะนำ:
  - <ข้อเสนอหรือ debt ที่ยังไม่ทำ>
```

## แหล่งข้อมูลสำหรับ "เหลืออะไร"

เวลาตอบคำถามกลุ่มนี้:

- "เหลืออะไร"
- "ยังไม่ได้ทำอะไร"
- "ยังไม่ได้ทำอะไรที่แนะนำ/เสนอมา"
- "อะไรยังค้าง"
- "วันนี้/เดือนนี้ทำอะไรไปบ้าง" แล้ว user ต้องการรู้งานต่อ

ต้องอ่านทั้ง 2 ไฟล์เสมอ:

1. `skill/day_dev_public/memory/PROGRESS.md`
2. `skill/day_dev_public/memory/WORK_LOG.md`

แล้วรวมรายการค้างจากทุกแหล่งนี้:

- `PROGRESS.md` หัวข้อ `ยังไม่ได้ทำ / ทำต่อ (Next)`
- `PROGRESS.md` หัวข้อ `Blocker / คำถามค้าง`
- `PROGRESS.md` หัวข้อ `คำแนะนำ (Recommendations)` บรรทัด `⏳ ยังไม่ได้ทำตาม`
- `WORK_LOG.md` หัวข้อ `เหลือ/แนะนำ` ของวันที่หรือเดือนที่เกี่ยวข้อง

**ห้ามตัดคำแนะนำที่ยังไม่ได้ทำตามออก** เพียงเพราะไม่ใช่งาน blocking เช่น "ยังไม่ release เพราะรอ user สั่ง"
ต้องบอกแยกเป็น recommendation backlog ให้เห็นชัด

รูปแบบตอบ default สำหรับคำถาม "เหลืออะไร":

```markdown
เหลือทั้งหมดตอนนี้:

## งานที่ยังเหลือ / Next
- <งานค้างจาก PROGRESS/WORK_LOG>

## คำแนะนำที่ยังไม่ได้ทำตาม
- <recommendation ที่ยังไม่ได้ทำตาม>

## Blocker / คำถามค้าง
- <ถ้าไม่มีให้บอกว่าไม่มี>
```

ถ้าไม่มีงานค้างจริง ให้ตอบชัดว่า "ไม่มีงานบังคับค้าง" แต่ยังต้องแสดงคำแนะนำที่ยังไม่ได้ทำตามถ้ามี

---

## เวลาถาม "วันนี้ทำอะไรบ้าง"

1. อ่าน `skill/day_dev_public/memory/WORK_LOG.md`
2. ใช้วันที่ปัจจุบันของ session แล้วเลือก entry ของวันนั้น
3. ถ้า `WORK_LOG.md` ยังไม่มี entry วันนี้ ให้ดู `PROGRESS.md` และบริบทล่าสุดประกอบ แล้วบอกข้อจำกัดชัดเจน
4. อ่าน `PROGRESS.md` ประกอบ เพื่อดึง `Next`, blocker และคำแนะนำ `⏳ ยังไม่ได้ทำตาม`
5. สรุปเป็นภาษาคน ไม่ใช่ dump log
6. ปิดท้ายด้วย "เหลือ/แนะนำ" ที่รวมงานค้างทั้งหมดและคำแนะนำที่ยังไม่ได้ทำตาม

รูปแบบตอบ default:

```markdown
วันนี้ทำไปหลักๆ:

1. <งาน/ระบบ>
   - ทำ: <สั้น>
   - ผลลัพธ์: <สั้น>
   - ตรวจแล้ว: <สั้น>

เหลือ/แนะนำ:
- งานที่ยังเหลือ: <ถ้าไม่มีให้บอกว่าไม่มีงานบังคับค้าง>
- คำแนะนำที่ยังไม่ได้ทำตาม: <ถ้ามีต้องใส่>
- Blocker/คำถามค้าง: <ถ้าไม่มีให้บอกว่าไม่มี>
```

ถ้า user บอกว่าจะเอาไปใส่สไลด์ ให้ปรับเป็น bullet สั้นแบบ slide-ready:

```markdown
สรุปผลงานประจำวันที่ <วันที่>
- <ระบบ>: <ทำอะไร> → <ผลลัพธ์>
- <ระบบ>: <ทำอะไร> → <ผลลัพธ์>
- ตรวจคุณภาพ: <lint/typecheck/build/test ที่ผ่าน>
- ประเด็นต่อเนื่อง: <งานค้างทั้งหมด + คำแนะนำที่ยังไม่ได้ทำตาม>
```

---

## เวลาถาม "เดือนนี้ทำอะไรบ้าง"

1. อ่าน `WORK_LOG.md`
2. เลือก entry ของเดือนปัจจุบันหรือเดือนที่ user ระบุ
3. รวมงานซ้ำเป็นกลุ่มตามระบบ/feature ไม่เรียงเป็นแชตย่อยทั้งหมด
4. อ่าน `PROGRESS.md` ประกอบ เพื่อดึงงานค้างล่าสุด, blocker และคำแนะนำ `⏳ ยังไม่ได้ทำตาม`
5. แยกผลลัพธ์กับงานค้างให้ชัด
6. หัวข้อ `เหลือ/แนะนำต่อ` ต้องรวมทั้งงานค้างและคำแนะนำที่ยังไม่ได้ทำตามทั้งหมดของเดือนนั้น/สถานะล่าสุด

รูปแบบตอบ default:

```markdown
สรุปงานเดือน <เดือน YYYY>

## งานที่ทำเสร็จ
- <ระบบ/feature>: <ผลลัพธ์>

## งานปรับปรุงคุณภาพ
- <lint/build/typecheck/performance/security/etc.>

## งานด้าน skill/process
- <เพิ่ม rule/flow อะไร>

## เหลือ/แนะนำต่อ
- งานที่ยังเหลือ: <debt/next>
- คำแนะนำที่ยังไม่ได้ทำตาม: <recommendation backlog>
- Blocker/คำถามค้าง: <ถ้ามี>
```

สำหรับสไลด์ ให้ใช้ 3-5 bullet ต่อสไลด์ และหลีกเลี่ยงชื่อไฟล์/ฟังก์ชันถ้าไม่จำเป็น

---

## กฎการเรียบเรียงสำหรับสไลด์

- เขียนแบบผลลัพธ์ก่อนรายละเอียด: "แก้ให้ build ตรวจ lint/typecheck ก่อน deploy" ดีกว่า "แก้ package.json"
- รวมงานจุกจิกเป็น theme เช่น "ยกระดับคุณภาพโค้ด React/TypeScript" แทน bullet แยกทุก lint rule
- เก็บคำสั่งตรวจที่สำคัญ เช่น `npm run lint`, `npm run typecheck`, `npm run build`
- ตัด path ยาว ชื่อไฟล์ย่อย และ implementation detail ถ้าไม่ได้ช่วยผู้ฟัง
- ถ้ามีข้อเสนอค้าง ให้เขียนเป็น "ประเด็นต่อเนื่อง" ไม่ปนกับงานที่เสร็จแล้ว

---

## Definition of Done เพิ่มเติม

เมื่อจบงานจริง:

1. อัปเดต `PROGRESS.md` สำหรับ resume
2. Append `WORK_LOG.md` สำหรับสรุปรายวัน/เดือน
3. ถ้า user ขอรายงาน/สไลด์ ใช้ [`report_skill.md`](./report_skill.md) ช่วยปรับภาษาให้ non-dev อ่านง่าย
